# Implementation Plan: Go API with GitHub Repository Validator

## Overview
Build a production-ready Go API using Gin framework with reusable logging/error handling and a demo endpoint to validate GitHub repositories.

## Phase 1: Project Foundation

### 1.1 Initialize Go Module
- Create `go.mod` with Go 1.21+ and Gin dependency
- Create `.env.example` with required environment variables
- Initialize basic project structure directories

### 1.2 Configuration Layer (`internal/config/`)
- Create `config.go` with Viper-based configuration
- Support environment variables: `PORT`, `ENVIRONMENT`

## Phase 2: Shared Utilities

### 2.1 Logging Package (`pkg/logger/`)
- Create `logger.go` with Zap/slog wrapper
- `Info()`, `Error()`, `Debug()`, `Warn()` functions
- Structured logging with fields support
- JSON output for production, human-readable for development

### 2.2 Error Handling (`pkg/errors/`)
- Create `errors.go` with sentinel errors:
  - `ErrNotFound`, `ErrUnauthorized`, `ErrBadRequest`, `ErrInternal`
  - `NewError(message string, code int)` for custom errors
  - `HTTPStatus(err error) int` helper

### 2.3 Response Helpers (`pkg/response/`)
- Create `response.go` with standardized JSON responses
- `OK()` - success response with data
- `Fail()` - error response with message
- Follow architecture.md format

## Phase 3: Demo Feature - GitHub Repository Validator

### 3.1 Model (`internal/model/`)
- Create `repository.go` with `Repository` struct:
  - `Owner`, `Name`, `Exists`, `URL` fields

### 3.2 DTO (`internal/dto/`)
- Create `repository_dto.go`:
  - `CheckRepositoryRequest` with `Owner`, `Repo` fields
  - `CheckRepositoryResponse` with `Exists`, `HTMLURL` fields

### 3.3 Service (`internal/service/`)
- Create `github_service.go`:
  - Interface for testability
  - `CheckRepositoryExists(owner, repo string) (*model.Repository, error)`
  - HTTP client to GitHub API (`https://api.github.com/repos/{owner}/{repo}`)
  - Timeout and retry handling

### 3.4 Handler (`internal/handler/`)
- Create `repository_handler.go`:
  - `CheckRepository()` endpoint handler
  - Accepts JSON body with owner/repo
  - Returns standardized response

## Phase 4: Entry Point & Routing

### 4.1 Main Entry (`cmd/api/main.go`)
- Initialize config, logger, router
- Wire all dependencies

### 4.2 Router Setup (`internal/router/router.go`)
- Create `GET /health` endpoint
- Create `POST /api/v1/repositories/check` endpoint
- Apply logging middleware globally

## Phase 5: Testing

### 5.1 Unit Tests
- `internal/service/github_service_test.go`:
  - Mock HTTP client for GitHub API tests
  - Test existing repo, non-existing repo, API errors

### 5.2 Handler Tests
- `internal/handler/repository_handler_test.go`:
  - Test with valid/invalid payloads
  - Test success and error responses
