# Implementation Plan: GitHub Issue Creation with Multi-Token Support

## 1. Goal
Add `CreateGitHubIssue` and `GetGitHubIssue` to the `GitHubService` interface:
- `CreateIssue(owner, repo, title, body string) (*model.Issue, error)`
- `GetIssue(owner, repo, issueNumber int) (*model.Issue, error)`

Both use per-repository API tokens pulled from a single `GITHUB_TOKENS` env-var JSON map so private repos can be queried. Full table-driven unit tests for the service and handler integration tests for both operations.

---

## 2. Design Questions & Decisions

### 2a. Secure Token Storage — **env-var map, no secrets manager dependency**

Per the no-assumptions rule, do not add any external secrets-manager library.  
Instead, expose a single `GITHUB_TOKENS` environment variable containing a JSON object keyed by `"owner/repo"`:

```bash
GITHUB_TOKENS='{"octocat/Hello-World":"ghp_token1","my-org/my-repo":"ghp_token2"}'
```

At startup, the service parses this map once. If the variable is absent or empty, **no tokens are available** and any attempt to call `CreateGitHubIssue` returns an error — this prevents accidental unauthenticated requests.

**Why this approach instead of individual env vars per repo?**  
A single JSON env var does not scale linearly with the number of repos; it keeps the config surface small regardless of how many repos the service supports. It also avoids hard-coding the env-var name per repo in Go source (a security concern — secrets shouldn't appear in code).

**Why not a CLI tool?**  
The only Go-level dependency we add is the stdlib `encoding/json` for parsing — already implicitly available. No `go install` step is required beyond `go mod tidy`.

---

### 2b. Per-Repository Token Selection

`CreateGitHubIssue` looks up `owner + "/" + repo` in the parsed token map. If found, it uses that token to authenticate with the GitHub API. If not found, it returns a clear error (`ErrTokenNotConfigured` sentinel).

---

### 2c. New Types / Enum

A new config struct must handle the token map, and the existing `Config` struct in `internal/config/config.go` should be extended to include `GitHubTokens`.

Actually, the token map is more naturally read and owned inside `github_service.go`'s constructor so it stays close to where it is used. Adding it to `Config` would make `Config` know about GitHub-specific concerns. The better split is:
- `Config` stays generic (Port, Environment).
- `githubService` constructor takes (`owner/repo → token`) map.

This means `NewGitHubService` will need an additional receiver or constructor function that accepts the map. Follow the same pattern as the existing `NewGitHubService()` — add an `opts` argument so the zero-argument constructor continues to work for existing callers (backwards compatible).

---

### 2d. `GitHubService` Interface Extension

Add two methods to the `GitHubService` interface:
```go
CreateIssue(owner, repo, title, body string) (*model.Issue, error)
GetIssue(owner, repo, issueNumber int) (*model.Issue, error)
```

All existing callers already satisfy the interface implicitly. The mock in `repository_handler_test.go` gets stub implementations for both new methods (they can return nothing/error since those tests only exercise `CheckRepositoryExists`).

## 4. Error Design

```go
var (
    ErrTokenNotConfigured = errors.New("no github token configured for the requested repository")
    ErrUnauthorized       = errors.New("unauthorized: invalid github token")
)
```

- `CreateIssue` returns `ErrTokenNotConfigured` when the token map is empty or has no entry for `owner/repo`.
- `GetIssue` returns `ErrTokenNotConfigured` for the same reason.
- `GetIssue` returns `ErrIssueNotFound` (a third sentinel) when GitHub returns HTTP 404 on `GET /repos/{owner}/{repo}/issues/{number}`.

All tested via `errors.Is`.

## 5. Test Strategy

### Service layer (`github_service_test.go`) — table-driven, `httptest.NewServer`

#### CreateIssue

| name | owner | repo | tokenMap | wantErr | wantErrIs |
|---|---|---|---|---|---|
| issue created | octocat | Hello-World | {owner/repo: token} | false | — |
| no tokens configured | octocat | Hello-World | nil | true | ErrTokenNotConfigured |
| no entry for repo | octocat | repo | {other/repo: token} | true | ErrTokenNotConfigured |
| GitHub 401 | octocat | Hello-World | {owner/repo: bad} | true | ErrUnauthorized |
| GitHub 500 | octocat | Hello-World | {owner/repo: token} | true | any |
| bad response body | octocat | Hello-World | {owner/repo: token} | true | any |

#### GetIssue

| name | owner | repo | number | tokenMap | wantErr | wantErrIs |
|---|---|---|---|---|---|---|
| issue found | octocat | Hello-World | 42 | {owner/repo: token} | false | — |
| no tokens configured | octocat | Hello-World | 42 | nil | true | ErrTokenNotConfigured |
| GitHub 404 → not found | octocat | Hello-World | 999 | {owner/repo: token} | true | ErrIssueNotFound |
| GitHub 401 | octocat | Hello-World | 42 | {owner/repo: bad} | true | ErrUnauthorized |

Extend `MockGitHubService` with stubs for both new methods.

- **`TestCreateIssue_OK`** — service returns issue → assert 200 + `CreateIssueResponse` JSON (`number`, `html_url`, `title`, `state`).
- **`TestCreateIssue_TokenNotConfigured`** — service returns `ErrTokenNotConfigured` → assert 500.
- **`TestCreateIssue_InvalidBody`** — malformed JSON body → assert 400.
- **`TestGetIssue_OK`** — service returns issue → assert 200 + `GetIssueResponse` JSON.
- **`TestGetIssue_TokenNotConfigured`** — service returns `ErrTokenNotConfigured` → assert 500.
- **`TestGetIssue_NotFound`** — service returns `ErrIssueNotFound` → assert 404.
- **`TestGetIssue_InvalidNumber`** — bad issue number in path → assert 400.

## 6. No-External-Library Constraint

Replying on `encoding/json` (stdlib) only for JSON. No secrets manager, no Vault, no cloud SDK. Single env-var parsing keeps the implementation self-contained and requires zero `go install` beyond the already-pinned dependencies in `go.mod`.
